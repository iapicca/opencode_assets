## What I do

- Read `tmp/implementation-plan.md` to understand what to implement
- Read specs from `.opencode/specs/` to understand project conventions
- Determine which MCP tools to use based on the programming language in `tech-stack.md`
- Generate code following the implementation steps precisely

> **Note**: This skill is read-only for `./tmp`. The `implementation-planner` agent writes to `./tmp`; this skill only reads from it.

## When to use me

- When you need to implement code from an implementation plan
- When you want to ensure code follows project specs and conventions

## Workflow

### 1. Read Implementation Plan

Use `tmp-file` skill to read `tmp/implementation-plan.md`:

1. Invoke `skill({ name: "tmp-file" })` to load the skill
2. Read `tmp/implementation-plan.md` fully
3. Extract:
   - **Task Summary**: What to implement
   - **Source Issue**: GitHub issue URL
   - **Issue Type**: Feature / Story / Task
   - **Working Branch**: The branch to create
   - **Target Branch**: Where this PR will merge into
   - **Files to Read**: Specific files to read before making changes
   - **Files to Change**: Files to modify
   - **New Files**: Files to create
   - **Implementation Steps**: Numbered steps to follow
   - **Tools to Use**: Which tools for each step
   - **Acceptance Criteria**: How to verify

### 2. Read Coding Standards

Read `coding-standards.md` from `.opencode/specs/`:

- Code style, patterns, naming conventions, testing requirements
- The pre-plan.md generated by pre-planner already summarizes architecture, tech-stack, and project-structure

### 3. Determine MCP Tools

Based on the implementation plan and pre-plan.md:

1. Check what language/framework is relevant for this task
2. Look for MCP tools matching that language/framework pattern (e.g., `mcp__<language>_*`, `mcp__<framework>_*`)
3. If MCP tools for the target language are available, use them for language-specific operations
4. If not available, use standard agent tools (edit, write, bash)

> The specific MCP tools available depend on the user's OpenCode configuration.

### 4. Create Working Branch

```bash
git checkout -b <working-branch-from-plan>
```

### 5. Execute Implementation

For each step in the Implementation Steps:

1. **Read** any required files using the `read` tool
2. **Edit** existing files using the `edit` tool
3. **Write** new files using the `write` tool
4. **Run** bash commands only if explicitly specified in the plan and within allowed permissions

### 6. Verify Against Acceptance Criteria

Check each acceptance criteria item from the plan:
- [ ] Review changes match what was specified
- [ ] Code follows conventions from coding-standards.md
- [ ] Files are in correct locations per project-structure.md
- [ ] Architecture patterns match architecture.md

### 7. Report Completion

Summarize:
- Files created/modified
- Implementation steps completed
- Any deviations from the plan (if needed)

## Code Generation Guidelines

### Always Follow
- Coding standards and conventions from `coding-standards.md`
- Implementation details from `tmp/implementation-plan.md`
- Pre-plan context from `pre-plan.md` (covers architecture, tech-stack, project-structure)

### Never
- Deviate from the implementation plan without user approval
- Create files outside the plan's scope
- Run build/test commands unless explicitly in the plan
- Modify files not listed in the plan

## MCP Tool Selection

When the tech stack specifies a language:

1. Check if MCP tools for that language are available in the current environment
2. If available, prefer MCP tools for language-specific operations (compiling, package management, etc.)
3. If not available, use standard bash with the appropriate commands
4. Always use agent tools (edit, write) for code modifications
